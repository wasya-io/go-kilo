# イベント駆動型アーキテクチャ移行計画

## 概要
現在のエディタをイベント駆動型アーキテクチャに移行し、より柔軟で拡張性の高い構造に改善する計画です。
この移行は段階的に行い、既存の機能を維持しながら新しいアーキテクチャを導入していきます。

## 移行の目的
- コンポーネント間の疎結合化
- 機能拡張の容易性向上
- テスト容易性の向上
- プラグイン機構の導入準備

## コマンドベースからイベントベースへの段階的移行計画

現在はコマンド層を全廃し、エディタの各種操作は入力を直接イベントに変換する方式に移行済みです。
以下の記述は過去の移行中のプロセスを記録したものです。

### 1. カーソル操作の移行（完了）
#### 理由
- 副作用が比較的少なく、状態変更が明確
- UI更新との連携が必要で、イベントベースの利点が活かせる
- 他の機能への影響が限定的

#### 移行手順
1. CursorEvent型の追加実装
   - Position、Direction情報を含む
   - イベントの優先度設定

2. カーソル移動コマンドの移行
   - 既存コマンドをラップしてイベント発行
   - 段階的な置き換え
   - UIのカーソル更新処理をイベントハンドラとして実装

3. スクロール連携
   - スクロール処理をイベントチェーンとして実装
   - カーソル移動に伴う自動スクロールの統合

#### カーソル操作のイベントフロー詳細
1. 現状の処理フロー
```
MoveCursorCommand.Execute()
└→ Editor.MoveCursor()
   └→ UI.MoveCursor()
      ├→ calculateNewCursorPosition() // 位置計算
      └→ SetCursor()                  // 位置更新
         └→ QueueUpdate(AreaCursor)   // 画面更新
```

2. イベントベース化後の処理フロー
```
MoveCursorCommand.Execute()
└→ Editor.MoveCursor()
   └→ eventManager.Publish(CursorEvent)
      └→ UI.handleCursorEvent()           // イベントハンドラ
         ├→ calculateNewCursorPosition()   // 位置計算（既存処理を再利用）
         └→ updateCursorPosition()         // 位置更新
            ├→ publishCursorUpdateEvent()  // 位置更新イベント発行
            └→ QueueUpdate(AreaCursor)     // 画面更新要求
```

3. カーソル関連イベントの種類
- CursorMoveEvent: カーソル移動要求
  - Direction: 移動方向（Up/Down/Left/Right）
  - Distance: 移動量
  - IsRelative: 相対移動か絶対移動か

- CursorUpdateEvent: カーソル位置更新通知
  - Position: 新しい位置（X, Y）
  - Source: 更新元（Move/Set/Scroll等）

4. イベントハンドラの責務
- UI.handleCursorEvent()
  - カーソル移動要求の処理
  - バッファ境界のチェック
  - 新しい位置の計算
  - マルチバイト文字の考慮

- UI.handleCursorUpdateEvent()
  - カーソル位置の更新
  - スクロール位置の調整
  - 画面更新の要求
  - 関連コンポーネントへの通知

5. 移行手順の詳細
- フェーズ1: イベント発行の導入
  - CursorEvent型の実装

## 現在のイベントキューイングと優先度制御アーキテクチャ (最新の実装状態)

以前の設計では「複数キューの分散（EventManagerの一時保管キュー、UIの保留キュー等）」「入力や描画ごとの複雑な優先順位制御（Priority Queue等）」などが課題として挙げられ構想されていました。しかし、その後のコードベースのリファクタリングにより**シンプルかつ効率的なアーキテクチャ**へ完全に統合・解決されました。

### 1. 単一のFIFOイベントバス (一元管理)
現在、イベントのキューイングは `app/entity/event/bus.go` が管理する単一のバッファ付きチャネル（キャパシティ100件）によって一元化されています。
- イベント（Save, Quit, Input, Cursor, Buffer, Refreshなど）は全て `eventBus.Publish()` 経由で等しく１つのキューへ投入されます。
- `bus.processEvents` ゴルーチンがこれを**FIFO（先入れ先出し）**で逐次ディスパッチします。
- これにより、キューの分散問題や、コンポーネント間の状態同期・順序保証の複雑さは完全に解消されました。

### 2. 優先度制御の代替としての「描画デバウンス遅延制御」
「描画更新より入力を優先する」といった複雑でオーバーヘッドの大きい優先度別キュー（Priority Queue）は導入せず、**「描画イベント（Refresh）のデバウンス処理（遅延評価）」** によって実質的な最適化を実現しています。

- **仕組み**: Controllerが `TypeRefresh` イベントを受け取ると、即座に画面を描画するのではなく、`time.AfterFunc` を用いて 16ms（約60FPS相当）のタイマーをセットします。
- **効果**: タイマー満了前に連続して入力イベントや追加のRefresh要求が発生した場合、タイマーがリセット・上書きされ続けます。結果として「重い描画処理がバッチ化・遅延され、軽い入力イベントの処理が優先される」という最適な状態を、最もシンプルなデータ構造（単一FIFO）のままで達成しています。

### 3. 今後の課題とスコープ
現在のFIFO＋デバウンス設計はデッドロックや優先度逆転のリスクがなく、安定しています。今後はこのシンプルで堅牢な機構をベースに、以下の機能を拡充していきます。

- **エラー伝播とリカバリー強化**: `EventError` 等の詳細なエラー構造を利用し、イベントチェーン内での部分的な失敗をより安全にハンドリング・リカバリする仕組みの整備。
- **監視機能の強化 (将来課題)**: 将来的に高度な拡張（プラグイン等）によりイベント量が急増した場合に備え、イベントバスのチャネル詰まりや処理レイテンシを計測する軽量なメトリクスの導入。
